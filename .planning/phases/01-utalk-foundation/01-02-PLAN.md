<objective>
**Create the UTalk message normalizer service that transforms UTalk webhook payloads into Ozion's internal message format.**

**Purpose:** UTalk webhooks use a unique form-encoded-style flat key format (`contact[number]`, `chat[body]`) instead of standard nested JSON. The normalizer handles this format and produces the same `NormalizedMessage` shape that Ozion already uses for Meta and Evolution messages — making UTalk messages interchangeable with other providers downstream.

**Output:** `server/services/utalk-normalizer.ts` (normalizer functions + type exports) and `tests/01-utalk-foundation.test.ts` (test coverage).
</objective>

    **Exports required:**

    1. `NormalizedMessage` interface:
    ```typescript
    export interface NormalizedMessage {
      from: string;
      fromName?: string;
      to: string;
      messageId: string;
      type: 'text' | 'image' | 'audio' | 'video' | 'document' | 'unknown';
      content: string;
      mediaUrl?: string;
      mediaMimeType?: string;
      direction: 'inbound' | 'outbound';
      timestamp: number;
      ack?: number;
      raw: any;
    }
    ```

    2. `StatusUpdate` interface:
    ```typescript
    export interface StatusUpdate {
      messageId: string;
      status: 'sent' | 'delivered' | 'read';
      timestamp: number;
    }
    ```

    3. `UTalkWebhookPayload` interface (with bracket-notation keys as string literals):
    ```typescript
    interface UTalkWebhookPayload {
      event?: string;
      token?: string;
      user?: string;
      operador?: string;
      'contact[number]'?: string;
      'contact[name]'?: string;
      'contact[server]'?: string;
      'chat[dtm]'?: string;
      'chat[uid]'?: string;
      'chat[dir]'?: string;
      'chat[type]'?: string;
      'chat[body]'?: string;
      ack?: string;
      muid?: string;
      id?: string;
    }
    ```

    4. Helper function `extractField(data: Record<string, any>, prefix: string, field: string): string`:
       - Returns `data[\`${prefix}[${field}]\`] || ''`
       - Example: `extractField(payload, 'contact', 'number')` returns `payload['contact[number]']`

    5. Helper function `mapMessageType(utalkType: string): NormalizedMessage['type']`:
       - Follow the type mapping table in RESEARCH.md:
         - `'chat'` → `'text'`
         - `'image'` → `'image'`
         - `'audio'` → `'audio'`
         - `'video'` → `'video'`
         - `'document'` → `'document'`
         - Default → `'unknown'`

    6. Helper function `mapDirection(dir: string): 'inbound' | 'outbound'`:
       - `'i'` → `'inbound'`
       - `'o'` → `'outbound'`
       - Default → `'inbound'`

    7. `normalizeIncomingMessage(payload: Record<string, any>): NormalizedMessage | null`:
       - Guard: if `payload.event !== 'chat'` return null
       - Extract fields using bracket-aware helper
       - Guard: if no `from` (contact[number]) or no `to` (user), return null
       - Build and return NormalizedMessage

    8. `normalizeStatusUpdate(payload: Record<string, any>): StatusUpdate | null`:
       - Guard: if `payload.event !== 'ack'` return null
       - Guard: if no `muid` or no `ack`, return null
       - Map ack values: `"1"` → `"sent"`, `"2"` → `"delivered"`, `"3"` → `"read"`, default → null (invalid)
       - Return StatusUpdate

    CRITICAL implementation details:
    - Never use `data.contact.number` syntax — UTalk uses flat `data['contact[number]']` keys
    - For `chat[body]`, pass through as-is (may contain text or base64 media data — Phase 2 handles distinction)
    - For `fromName`, use `contact[name]` if available, fall back to `from` phone number
    - For `content` when type is text, use `chat[body]`; for media types, use `chat[body]` (may be base64 or caption)
    - For `timestamp`, parse `chat[dtm]` as integer; if NaN/0, use `Math.floor(Date.now() / 1000)`
    - Log a `console.warn(\`⚠️ UTalk unknown message type: \${utalkType}\`)` for unmapped types
  </action>
